# tvbridge - Alexa turns the Samsung TV on/off

Lets Alexa power the Samsung TV **on** and **off** by voice, on a VM, without a
SmartThings hub.

```
"Alexa, allume la télé"  ->  Wake-on-LAN burst        (wol.py)
"Alexa, éteins la télé"  ->  KEY_POWER over WebSocket  (tvoff.py)
```

## Why this exists

The TV drops off the network in standby. Without a SmartThings hub, Alexa's
"on" command has to go through Samsung's cloud, which cannot reach a TV that is
offline. So turning the TV **on** by voice was structurally impossible, whatever
the TV settings.

The bridge works around it: an always-on VM sends a **Wake-on-LAN** magic packet
on the LAN (the TV's wired NIC answers WoL even when its cloud link is down).
`fauxmo` makes the VM look like a **WeMo plug** named `Télé`, which Alexa
discovers natively (no skill, no SmartThings).

## Components

| File | Role |
|------|------|
| `wol.py` | Sends a burst of WoL magic packets. LAN IP/broadcast auto-detected from the route to the TV (never leaks onto the VPN). |
| `tvoff.py` | Sends `KEY_POWER` via the Samsung WebSocket remote API. Pairing token stored in `tv_token.txt`. |
| `fauxmo.json` | Generated by `install.sh`. WeMo device `Télé`, on/off commands, `use_fake_state`. |
| `tvbridge.service` | Generated by `install.sh` into `/etc/systemd/system/`. |
| `install.sh` | Idempotent installer/reinstaller. |
| `requirements.txt` | Pinned deps (`fauxmo`, `samsungtvws`). |

Runtime target on the VM: `~/tvbridge/` (venv + deployed scripts + token).

## Install / reinstall (after a VM rebuild)

Run **on the VM**, as the service user (needs sudo):

```bash
git clone <this-repo> && cd free-iptv-app/server/tvbridge
./install.sh
```

Then, once per Alexa account:

```
"Alexa, découvre mes appareils"
```

### Pairing token

`tv_token.txt` is **not** committed (it is a per-device secret). Two ways to get it:

- **Restore** a token you kept: `echo <TOKEN> > ~/tvbridge/tv_token.txt`
  (as long as the TV was not factory-reset, the old token still works).
- **Re-pair**: say "Alexa, éteins la télé" once and accept the "Allow this
  device?" popup on the TV within 30 s. The token is then saved automatically.

## Hardcoded values

- TV MAC: `d4:9d:c0:5d:d9:c9` (in `wol.py`)
- TV IP: `192.168.1.241` (in `wol.py` and `tvoff.py`)
- WeMo device name: `Télé`, port `12340`

The VM LAN IP, broadcast and interface are **auto-detected**, so a new DHCP lease
or a renamed interface after a rebuild is handled automatically.

## The VPN gotcha (important)

The VM's default route is a VPN (`wg0`). Two things must stay on the LAN:

1. **WoL broadcast** - `wol.py` binds the socket to the detected LAN IP.
2. **SSDP discovery multicast** - `fauxmo` joins `239.255.255.250` with
   `INADDR_ANY`, which would follow the VPN default route and make Alexa unable
   to discover the device. The service's `ExecStartPre` adds a
   `224.0.0.0/4 dev <lan-iface>` route so multicast stays on the LAN.

## Troubleshooting

```bash
systemctl status tvbridge            # service up?
journalctl -u tvbridge -f            # live logs (WoL / KEY_POWER lines)
ip route get 239.255.255.250         # must be 'dev <lan-iface>', NOT wg0
```

- **"Alexa says it can't find the device"**: check the multicast route above,
  then re-run discovery.
- **"allume" says OK but TV stays off**: the TV was in deep sleep and missed the
  packet; the 10-packet burst normally covers it. Check `journalctl` shows the
  WoL line, and that the TV's *Settings > General > Network > Expert > Power On
  with Mobile* is still enabled.
- **"éteins" fails with a timeout**: the token is missing/expired, re-pair (see
  above).
